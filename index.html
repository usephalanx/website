How It Works

Four steps from command
to production

Define your formation once. Phalanx handles the coordination. You keep the authority.

01

Configure

Define agent skills, guardrails, and approval gates in YAML. No code changes needed to adapt Phalanx to your stack.

configs/team.yaml
02

Delegate

Send a work order from Slack. Describe the task in plain English. Commander decomposes it into an ordered task plan.

/phalanx build "Add JWT auth"
03

Approve

Review the plan before a single line of code is written. Reject, revise, or approve with one click in Slack.

✓ Approve Plan
04

Ship

Agents build, test, review, and security-scan. You approve the final PR. Nothing merges without your sign-off.

The Formation

Each agent has a clear mandate.
None with unchecked authority.

Not one model doing everything. Specialized agents with distinct roles, skills, and review criteria — coordinated by a state machine.

DRelease
C
Commander

Coordinates the entire run. Accepts work orders, manages state transitions, and orchestrates handoffs between agents.

P
Planner

Decomposes work into ordered tasks with file paths, function names, and test cases. Precise enough that no questions need to be asked.

B
Builder

Writes code, creates files, and commits changes to a feature branch. Tasks run in parallel where dependencies allow.

R
Reviewer

Reviews all commits for code quality, correctness, and adherence to project patterns. Scores and flags issues before QA.

Q
QA

Runs the test suite, measures coverage delta, and verifies all tests pass. Blocks the run if coverage drops.

S
Security

Scans for vulnerabilities, secret exposure, injection risks, and trust-boundary violations. Blocks on any critical finding.

D
Release

Opens the PR, tags the release, and runs health checks. The human approves the final merge — then it ships.

You Stay In Control

Three gates where you decide.
Agents can't skip them.

Autonomous doesn't mean unsupervised. Every run waits for your explicit approval at three mandatory checkpoints before advancing.

1
Before any code is written

Plan Approval

Review the full implementation plan — every task, every file, every dependency — before Commander dispatches a single agent. Reject and revise or approve to proceed.

AWAITING_PLAN_APPROVAL
2
After build, review, QA & security

Ship Approval

Once all agents have completed their work, you see the full results: code quality score, test coverage, and security scan — before anything is merged.

AWAITING_SHIP_APPROVAL
3
Before production release

Release Approval

Final sign-off before the tagged release goes out. The run stays in RELEASE_PREP until you explicitly approve. Nothing ships without you.

AWAITING_RELEASE_APPROVAL

Workflow

A 16-state machine,
not ad-hoc prompts

Every run follows a deterministic path. State transitions are explicit, auditable, and impossible to skip.

INTAKE→ RESEARCHING→ PLANNING→ AWAITING_PLAN_APPROVAL→ EXECUTING→ VERIFYING→ AWAITING_SHIP_APPROVAL→ READY_TO_MERGE→ MERGED→ RELEASE_PREP→ AWAITING_RELEASE_APPROVAL→ SHIPPED

Capabilities

Built for teams that ship

Every feature exists because we needed it building real production software.

Config-Driven Workflows

Define agent behavior, approval gates, and skill sets in YAML. Change your process without touching Python.

Human Approval Gates

Three mandatory checkpoints: plan, ship, and release. Nothing moves forward without your explicit sign-off.

IC-Level Agent Skills

Agents have configurable skill registries — language expertise, framework knowledge, testing patterns. Match to your stack.

Project Memory

Postgres + pgvector stores prior runs, decisions, and code context. Agents learn from project history, not just the current prompt.

Security by Default

Dedicated Security agent runs on every build. Blocks the pipeline on critical vulnerabilities before anything merges.

Deploy Anywhere

Docker Compose for local. Single-command deploy to any cloud. Your infrastructure, your data, your rules.
