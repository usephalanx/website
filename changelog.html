v1.4.0 2026-04-25 Feature Agents Infrastructure

CI Fixer v3 — a 4-agent DAG with on-the-fly sandbox provisioning, iteration on cascading failures, and the first agent commit on a third-party OSS repo (usephalanx/humanize#2). v2 keeps running for every existing repo; v3 is opt-in per ci_integrations.cifixer_version='v3' with automatic fallback to v2 if dispatch fails.

  • Multi-agent DAG: SRE Setup → Tech Lead → Engineer → SRE Verify. Each agent has a single responsibility and writes a structured artifact downstream agents read. Tech Lead investigates with read-only tools (GPT-5.4); Engineer applies the fix and verifies in sandbox (Sonnet); SRE handles env provisioning and full-CI mimicry. Commander orchestrates state, never reads code.
  • On-the-fly sandbox provisioning. v3 boots a fresh container from a minimal base image per run (e.g. python:3.11-slim) and installs deps the way the repo asks — no more pre-warmed pool with a year-stale ruff. env_detector reads pyproject.toml, .github/workflows/*.yml and emits an EnvSpec the provisioner consumes.
  • Iteration loop on cascading failures. When SRE's full-CI verify reports new failures after an Engineer commit, Commander appends another [techlead, engineer, sre_verify] round (capped at 3 iterations) and feeds the prior failures into the next Tech Lead's prompt. The verification gate held on every iteration in the canary.
  • First commit on an external OSS repo. 75b624a on usephalanx/humanize PR #2 — a one-line E501 fix on src/humanize/number.py, sandbox-verified before push. See the public scorecard's real-world section for the full pipeline trace.
  • Two-tier integration harness. tests/integration/v3_harness/ (Tier-1, 51 tests, no Postgres/Docker) covers per-language env_detector contracts (Python full, TS/JS/Java/C# Phase-1) and parser robustness across LLM output shapes. tests/integration/v3_harness_t2/ (Tier-2, 13 tests) catches provider-shape and signature-shadowing regressions via real Postgres + a schema validator that mimics the OpenAI Responses API. Combined ~2s feedback loop.
  • Tech Lead prompt v3. Two policy refinements from the canary: (1) failing_command must be the narrowest command that re-runs the failing check, NOT the CI wrapper (no prek run, make ci, etc.) — isolates the fix from sandbox env quirks; (2) when the sandbox can't run CI as configured (e.g. uv missing), set confidence=0.0 and escalate — never edit .github/workflows/, tox.ini, etc. to dodge the env mismatch.
  • Real-world section on the public scorecard. scorecard.html now renders external OSS commits in a distinct section above the testbed grid. Generator emits real_world_runs[] alongside the existing cells[]; section is hidden until at least one real-world run lands.
  • Database migration. Two columns added (zero-downtime, both with server defaults): work_orders.work_order_type (distinguishes Slack build from CI ci_fix) and ci_integrations.cifixer_version (per-repo v2/v3 selector). Migration 20260423_0001.
  • Honest about the rough edges. Eight bugs surfaced during the canary — six were "v3 forgot something v2's bootstrap glue does" (Celery include, task lifecycle persistence, Sonnet wiring, OpenAI tool_result shape, _audit signature, fix_spec parser); two were prompt issues (the rules above). The harness now catches 6 of 8 locally pre-deploy. The full retro is in the repo at docs/ci-fixer-v3-canary-retro.md.
v1.3.82 2026-04-11 Feature Agents

Workspace isolation per run: every build gets a clean, isolated directory. QA now reads a machine-generated QA.md recipe instead of recursively scanning the codebase. All agents resolve the workspace from a single source of truth on the Run record.

  • Isolated workspaces per run. Each run gets its own directory at /tmp/forge-repos/{slug}-{run_id[:8]}/. First builder task always clones fresh; subsequent tasks in the same run commit incrementally. Eliminates cross-run file contamination that caused QA to pick up stale test results.
  • QA.md recipe-driven testing. The final builder task generates a QA.md YAML file — stack, test runner, install steps, coverage threshold — using GPT. QA reads it directly instead of doing an LLM recursive scan of the workspace. React/Vite apps now correctly receive peer dependency install steps (@testing-library/jest-dom, jsdom, vitest) before tests run.
  • Uniform workspace resolution across all agents. Builder, QA, Security, and Reviewer all read Run.workspace_path from the database — one column, set once by the first builder task. No agent invents its own path formula.
  • SRE slug collision fix. Running the same app title twice no longer triggers a demos_slug_key unique violation. SRE now appends -{run_id[:8]} when a slug is already taken by a different run.
  • Database migration. Added workspace_path column to the runs table (nullable Text). Migration 20260411_0001.
v1.3.81 2026-04-10 Agents Infrastructure

SRE agent now runs in its own isolated Celery worker with full demo infrastructure wiring. Every completed run auto-deploys a live containerized demo at demo.usephalanx.com/{slug}.

  • Isolated SRE Celery worker. SRE tasks now run in a dedicated phalanx-sre-worker container, separate from the main worker. Prevents long-running Docker builds and image pushes from blocking builder and QA tasks.
  • Demo portal wiring. SRE agent registers each deployed app in the demos table. The demo portal at demo.usephalanx.com serves all live demo containers with per-slug nginx routing and automatic HTTPS.
  • LLM-generated Dockerfile + nginx config. SRE uses GPT-4.1 to generate a production-ready Dockerfile for each app based on the detected stack (Python/FastAPI, Node/Vite, etc.), then builds and deploys it to the demo server in parallel with infra prep.
v1.3.80 2026-04-09 Feature Agents

GPT-4.1 promoted to the primary reasoning model for planning and review tasks. Node.js added to the worker image for React/Vite builds. Vague task enrichment prevents the builder from acting on under-specified prompts.

  • GPT-4.1 for reasoning agents. Commander, Planner, and Reviewer switch to gpt-4.1 for plan generation and code review. Builder retains Claude for code generation. Both models run in the same pipeline per task.
  • Node.js in production worker. nodejs and npm installed in the worker image so React, Vite, and Next.js workspaces can be built and tested inside the container without a separate Node runtime.
  • Prompt enricher agent. Vague or ambiguous work orders are automatically enriched before planning. If the intent can't be resolved with high confidence, the run pauses and asks for clarification rather than proceeding with a hallucinated interpretation.
v1.3.79 2026-04-08 Agents Infrastructure

SRE agent ships LLM-driven Dockerfile generation and parallel infrastructure preparation. Slack progress board with five-phase narrative threading.

  • SRE agent: autonomous deployment. After release, the SRE agent generates a Dockerfile for the built app, builds a Docker image, pushes it to the demo server, and configures nginx — entirely autonomously. Each run results in a live, publicly accessible demo URL.
  • Slack threaded progress narrative. The Slack bot now posts a structured progress thread across five phases (plan, build, review, QA/security, ship). Each phase is a reply in the thread so the main channel stays clean and the thread tells the full story of the run.
  • Retry hardening + stale task watchdog. Celery tasks that fail transiently are retried with exponential backoff. A background watchdog detects tasks stuck in IN_PROGRESS beyond their soft time limit and marks them failed with a timeout reason.
v1.3.78 2026-04-07 Feature Infrastructure

CI fixer agent added to the pipeline. Agent reasoning traces and run timeline API. CI webhook ingestion for GitHub, Buildkite, CircleCI, and Jenkins.

  • CI fixer agent. When a CI run fails after merge, the CI fixer agent is automatically dispatched. It reads the failure logs, diagnoses the root cause, writes a targeted fix, and opens a follow-up PR — without human intervention.
  • Agent traces API. Every agent decision — prompts sent, reasoning steps, file writes, tool calls — is recorded as a structured trace. Accessible at GET /runs/{run_id}/trace. A timeline portal UI ships with this release for visual inspection.
  • CI webhook ingestion. Phalanx can now receive webhooks from GitHub Actions, Buildkite, CircleCI, and Jenkins at POST /webhook/{provider}. CI failures automatically trigger the CI fixer pipeline.
  • Demo portal. demo.usephalanx.com serves every app deployed by the SRE agent. The portal lists all live demos with run metadata, stack info, and direct links.
v1.3.14 2026-03-20 Initial OSS Release

First public release of Phalanx under the usephalanx org. Core multi-agent pipeline with Commander, Planner, Builder, Reviewer, QA, Security, and Release agents. Human approval gates at plan, ship, and release. Self-hostable via Docker Compose.

  • Core agent pipeline. Seven specialized agents — Commander, Planner, Builder, Reviewer, QA, Security, Release — coordinated by a 16-state deterministic run machine. Each stage hands off via Celery task queues.
  • Human approval gates. Three mandatory gates: plan approval (before any code is written), ship approval (before merge), and release approval (before deploy). Non-skippable in code.
  • Slack gateway. Socket-mode Bolt app receives /phalanx build commands, posts plan approvals as interactive Block Kit messages, and threads all progress updates.
  • Security pipeline. Every build runs detect-secrets, bandit, pip-audit, and optional Trivy before the ship gate. Critical findings block the run.
  • Self-hosted Docker Compose stack. PostgreSQL + pgvector, Redis, FastAPI gateway, Celery worker fleet. Single ./deploy.sh ships to any Linux VM. No vendor lock-in.
